## 為何選 EMQX？

### 候選方案比較

| 方案 | 語言 | 單節點並發 | 叢集 | License |
|---|---|---|---|---|
| **EMQX 5** | Erlang/OTP | ~100 萬 | 內建 Raft | Apache 2.0（CE） |
| Mosquitto | C | ~20 萬 | 需外掛 | EPL 2.0 |
| HiveMQ CE | Java | ~10 萬 | 付費 | Apache 2.0（CE）|
| VerneMQ | Erlang | ~100 萬 | 需設定 | Apache 2.0 |
| AWS IoT Core | 雲服務 | 無上限 | 自動 | 按用量計費 |

### 選 EMQX 的理由

1. **Erlang/OTP 天生高並發**：Actor model，每個 MQTT 連線一個輕量 process，互不阻塞
2. **官方 Docker image 完善**：開箱即用，healthcheck 設計良好
3. **管理後台豐富**：內建 Web UI 可監控連線數、訊息速率、主題訂閱圖
4. **Rule Engine**：EMQX 5 Enterprise 有無 SQL 規則引擎可轉發到 Kafka/InfluxDB，CE 版需自己訂閱
5. **可觀測性**：Prometheus metrics、OpenTelemetry 支援

### EMQX 的限制

- CE 版不含 Rule Engine 和部分企業功能
- Erlang 技術棧不常見，自訂 hook 需要寫 Erlang/Elixir
- 記憶體用量相對較高（每連線約 50KB）

## MQTT QoS 策略

MQTT 定義三種服務品質等級：

### QoS 0 — At Most Once（最多一次）

```
Sender ──PUBLISH──→ Broker ──PUBLISH──→ Subscriber
        （不確認）          （不確認）
```

- **特性**：發送後不等 ACK，封包可能遺失
- **適用場景**：高頻率取樣且偶爾掉包無影響（如溫度每秒更新）
- **本專案**：模擬器使用 **QoS 1**，確保資料進 InfluxDB

### QoS 1 — At Least Once（至少一次）

```
Sender ──PUBLISH──→ Broker ──PUBACK──→ Sender
        Broker ──PUBLISH──→ Subscriber ──PUBACK──→ Broker
```

- **特性**：確保到達，但可能重複
- **適用場景**：感測器資料（重複寫 InfluxDB 無害，時間戳相同會覆蓋）
- **本專案**：感測器上行資料使用 **QoS 1**

### QoS 2 — Exactly Once（恰好一次）

```
Sender ──PUBLISH──→ Broker ──PUBREC──→ Sender ──PUBREL──→ Broker ──PUBCOMP──→ Sender
```

- **特性**：四步握手，保證恰好一次，最高延遲
- **適用場景**：控制指令（不能重複執行，如「開機」送兩次會有問題）
- **本專案**：`control.py` 下指令使用 **QoS 2**

### 本專案 QoS 矩陣

| 方向 | 資料類型 | QoS | 理由 |
|---|---|---|---|
| 設備 → Broker | 感測器資料 | 1 | 確保進庫，重複無害 |
| Broker → Backend | 感測器訂閱 | 1 | 同上 |
| Backend → Broker | 控制指令 | 2 | 指令不能重複執行 |
| Broker → 設備 | 控制指令 | 2 | 同上 |

## InfluxDB vs TimescaleDB vs ClickHouse

### 候選方案

| 方案 | 基礎 | 查詢語言 | 適合場景 |
|---|---|---|---|
| **InfluxDB 2** | 自研 TSM 引擎 | Flux | IoT / Metrics，開箱即用 |
| TimescaleDB | PostgreSQL 擴充 | SQL | 熟 SQL 的團隊，混合查詢 |
| ClickHouse | 列存 OLAP | SQL | 海量歷史分析，億行級 |
| VictoriaMetrics | 自研 | MetricsQL | Prometheus 替代，超高壓縮比 |

### 選 InfluxDB 的理由

1. **IoT 原生設計**：Tag/Field 模型天生對應感測器概念（tag = 標籤維度，field = 量測值）
2. **寫入效能**：批次寫入可達每秒數十萬點
3. **Flux 語言**：函數式管道，aggregateWindow、pivot、join 直觀
4. **Docker image 整合良好**：環境變數直接初始化 bucket/org/token

### InfluxDB 的限制

- Flux 語言學習曲線較陡（和 SQL 思維不同）
- InfluxDB 3 已改用 Apache Arrow + SQL，2.x 是過渡期
- JOIN 跨 measurement 效能不佳

### 何時換 TimescaleDB？

- 已有 PostgreSQL 技術棧，不想多一個資料庫
- 需要同時查詢時序資料和關聯資料（如 JOIN 設備資料）
- 資料量不超過數億筆

### 何時換 ClickHouse？

- 歷史資料超過 1 億筆需要快速聚合
- 需要複雜的 OLAP 分析（如多維度 GROUP BY）

## FastAPI vs Django vs Node.js

| 面向 | FastAPI | Django | Express/Fastify |
|---|---|---|---|
| 效能 | 極高（async） | 中（sync/async 混用）| 高（V8 event loop）|
| MQTT 整合 | 自然（執行緒+async）| 笨重（Celery 繞路）| 原生（mqtt.js）|
| WebSocket | 內建支援 | Channels（複雜）| socket.io |
| Schema 驗證 | Pydantic v2 | DRF Serializers | zod / joi |
| 自動 API 文件 | 內建 Swagger | 需外掛 | 需外掛 |
| 部署 | uvicorn / gunicorn | gunicorn | pm2 |

### 選 FastAPI 的理由

1. **Pydantic v2**：自動驗證、序列化、文件生成，一石三鳥
2. **async/await**：WebSocket 和 HTTP 共用一個 event loop，無縫整合
3. **型別提示**：IDE 自動補全，重構安全
4. **`/docs` 自動生成**：開發者接手時直接看 Swagger，不需額外文件

## WebSocket 跨執行緒推播設計

### 問題

MQTT 的 `on_message` 回調運行在 **paho 的背景執行緒**，而 WebSocket 的 `send()` 是 **asyncio coroutine**，不能直接在執行緒中 await。

### 錯誤做法

```python
# ❌ 在 MQTT 執行緒中直接 await — 會報 RuntimeError
await ws_manager.broadcast(data)

# ❌ asyncio.run() — 會建立新的 event loop，與 FastAPI 的 loop 不同
asyncio.run(ws_manager.broadcast(data))
```

### 正確做法

```python
# ws_manager.py
class ConnectionManager:
    def set_loop(self, loop: asyncio.AbstractEventLoop) -> None:
        self._loop = loop   # ← FastAPI lifespan 時存下來

    def broadcast_from_thread(self, data: dict) -> None:
        # 跨執行緒投遞 coroutine 到 FastAPI 的 event loop
        asyncio.run_coroutine_threadsafe(
            self.broadcast(data),
            self._loop
        )
```

```python
# main.py lifespan
loop = asyncio.get_running_loop()
ws_manager.set_loop(loop)    # ← 必須在啟動 MQTT 執行緒之前
```

**關鍵**：`asyncio.run_coroutine_threadsafe()` 是 thread-safe 的，它把 coroutine 包裝成 `Future` 投遞到目標 loop 的執行佇列，MQTT 執行緒本身繼續執行不阻塞。

## 多租戶隔離的三種層次

| 隔離層次 | 說明 | 成本 | 安全性 |
|---|---|---|---|
| **Row-level**（本專案）| 共用 DB，每筆資料帶 customer_id | 低 | 中（需嚴格過濾）|
| **Schema-level** | 每個 customer 一個 PG schema | 中 | 高 |
| **Database-level** | 每個 customer 獨立資料庫實例 | 高 | 最高 |

本專案使用 **Row-level**，適合中小規模（< 1000 客戶）。

擴展建議：若客戶數超過 1000，可考慮 InfluxDB 改用多 bucket（每客戶一個 bucket），PG 改用 Row-Level Security。

## IoT 安全性考量

### 本機測試（目前實作）

- MQTT 無 TLS、無認證（EMQX 開放匿名連線）
- 所有服務使用預設密碼

### 生產環境最小安全基線

1. **MQTT 認證**：在 EMQX 建立 Username/Password，IoT Gateway 帶帳密連線
2. **MQTT TLS**：Gateway 連 EMQX 8883 port，使用 Let's Encrypt 或自簽憑證
3. **API 認證**：加 JWT Bearer token（FastAPI 搭 `python-jose`）
4. **網路隔離**：EMQX / PG / InfluxDB 不對外暴露，只有 backend 暴露 8000
5. **Secret 管理**：環境變數透過 Docker Secrets 或 HashiCorp Vault 注入

### 控制指令安全

```json
{
  "action": "turn_off",
  "request_id": "uuid-v4",
  "expire_at": 1750000010
}
```

- `request_id`：設備去重，防止重放攻擊
- `expire_at`：設備離線重連後應忽略過期指令
- 生產應加 HMAC 簽章驗證指令來源

## 水平擴展路線圖

### 第一階段：垂直擴展（< 1000 設備）

本專案現況，單機 Docker Compose 即可。

### 第二階段：EMQX 叢集（1000 ～ 10 萬設備）

```
               Load Balancer (L4 TCP)
              /          |          \
         EMQX-1        EMQX-2        EMQX-3
              \          |          /
              Raft Consensus Cluster
```

- EMQX 節點間自動同步 session 和訂閱
- Backend 只需連任一節點

### 第三階段：後端水平擴展（> 10 萬設備）

引入 Kafka 作為 MQTT → 後端的 Buffer：

```
EMQX ──Rule Engine──→ Kafka ──Consumer Group──→ Backend workers (N 台)
                                                        │
                                                   InfluxDB Cluster
                                                   PostgreSQL (Read Replicas)
```

- Backend 無狀態化（WebSocket 狀態移到 Redis Pub/Sub）
- InfluxDB 改用 InfluxDB Clustered 或 Grafana Mimir

### 第四階段：Serverless / 雲原生

- AWS IoT Core 取代 EMQX（按連線計費）
- AWS Timestream 取代 InfluxDB
- Aurora PostgreSQL 取代 PG
- ElastiCache 取代 Redis

## 感測器時間戳的重要性

### 問題：用伺服器時間還是設備時間？

```
設備 → 量測值：-22.3°C，時間戳：12:00:00
4G 網路延遲 3 秒
伺服器收到：12:00:03
```

**若用伺服器時間**：資料點標記在 12:00:03，但實際是 12:00:00 量測的  
**若用設備時間**：資料點標記在 12:00:00，正確反映實際量測時刻

### 本專案做法

payload 帶 `ts`（UNIX timestamp），MQTT subscriber 用設備時間寫入 InfluxDB：

```python
.time(ts, WritePrecision.S)  # ← 設備時間
```

### 設備時鐘漂移問題

若設備時鐘不準（IoT 設備常見），資料點可能「早到」或「晚到」。生產環境建議：
1. Gateway 使用 NTP 對時
2. 後端拒絕時間戳偏差超過 ±5 分鐘的資料點
3. InfluxDB 若收到過去時間戳的資料，會正確插入而非覆蓋（只要時間戳不同）
