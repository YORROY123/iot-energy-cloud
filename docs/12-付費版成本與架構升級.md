# 12 — 付費版成本試算與架構升級規劃

這份文件給「要把 Demo 變成正式商用服務」時參考：**各規模的月成本試算、需要修改的程式、以及對應的架構演進**。  
分成三個階段（小型試營運 → 中型商用 → 大型多租戶），可依實際客戶數逐步升級。

## 階段一：小型試營運（≤100 台設備 / 數百人觀看）

讓服務「不休眠、資料留久一點、有基本安全」，但仍用雲端託管省維運。

| 元件 | 升級為 | 規格 | 月成本（USD）|
|---|---|---|---|
| 後端 | Render Starter | 0.5 CPU / 512MB，不休眠 | 7 |
| 關聯式資料庫 | Render PostgreSQL Basic | 1GB，不會 90 天到期 | 7 |
| 時序資料庫 | InfluxDB Cloud（用量計費）| 保留 30～90 天 | ~10～20 |
| 快取 | Upstash Pay-as-you-go | 超過免費額度部分 | ~0～5 |
| MQTT Broker | HiveMQ Cloud Serverless | 100 連線內 | ~0～10 |
| 看板 | GitHub Pages（維持免費）| | 0 |
| **合計** | | | **約 US$30～50 / 月** |

**承載力**：同時數百人觀看、上百台設備穩定運作。

## 階段二：中型商用（≤1000 台設備 / 數千人）

開始需要「專屬 broker、後端可多開、資料庫獨立」。建議搬到單一雲（AWS / GCP）。

| 元件 | 方案 | 規格 | 月成本（USD）|
|---|---|---|---|
| 後端 | 雲端 VM × 2（負載平衡）| 2 vCPU / 4GB each | ~60～100 |
| MQTT Broker | 自架 EMQX（1～3 節點）| VM 2 vCPU / 4GB | ~40～120 |
| 時序資料庫 | InfluxDB Cloud 付費 或自架 | 保留 1 年 | ~50～150 |
| 關聯式資料庫 | 雲端託管 PostgreSQL | 2 vCPU / 8GB | ~50～100 |
| 快取 | 雲端託管 Redis | 1GB | ~15～30 |
| 看板 / CDN | 物件儲存 + CDN | | ~5～15 |
| 負載平衡 | 雲端 LB | | ~20 |
| **合計** | | | **約 US$250～500 / 月** |

**承載力**：上千台設備、數千人同時觀看。

## 階段三：大型多租戶（上萬台設備）

進入「水平擴展 + 高可用 + 資料分層」階段。

| 元件 | 方案 | 月成本（USD）|
|---|---|---|
| 後端 | Kubernetes 叢集自動擴縮（5～20 pod）| ~300～800 |
| MQTT Broker | EMQX 叢集（3～5 節點，含 Enterprise 規則引擎）| ~300～1000 |
| 訊息緩衝 | Kafka（解耦 broker 與後端）| ~150～400 |
| 時序資料庫 | InfluxDB Clustered 或 自架叢集 | ~300～800 |
| 關聯式資料庫 | 雲端 PostgreSQL（含讀取副本）| ~200～500 |
| 快取 | Redis 叢集 | ~80～200 |
| 監控 | Prometheus + Grafana | ~50～100 |
| **合計** | | **約 US$1,500～4,000 / 月** |

**承載力**：上萬台設備、上萬人同時觀看、99.9% 可用性。

## 需要修改的程式清單

從免費 Demo 升級到正式版，需要動的地方（依優先序）：

### 🔴 必改（安全與正確性）

| 項目 | 現況 | 要改成 | 涉及檔案 |
|---|---|---|---|
| MQTT broker | 公開 HiveMQ | 自架 EMQX + 帳密 + TLS | `docker-compose.yml`、環境變數 `MQTT_HOST` |
| API 認證 | 全開無驗證 | 加 JWT / API Key | `backend/main.py`、各 `routers/` |
| CORS | `allow_origins=["*"]` | 限定你的前端網域 | `backend/main.py` |
| 後台保護 | 僅踢人需金鑰 | 整個 `/admin/*` 加登入 | `backend/main.py` |
| 模擬資料 | 後端內建發布器 | 關閉 `RUN_SIMULATOR`，接真實 Gateway | 環境變數 |
| 機密管理 | 環境變數明文 | Secrets Manager / Vault | 部署設定 |

### 🟡 建議改（效能與可靠度）

| 項目 | 現況 | 要改成 | 涉及檔案 |
|---|---|---|---|
| InfluxDB 寫入 | 每筆同步寫 | 批次 / 非同步寫入 | `backend/mqtt_subscriber.py` |
| 設備狀態更新 | 每筆都 UPDATE PG | 節流（每 N 秒一次）| `backend/mqtt_subscriber.py` |
| WebSocket 狀態 | 存在單一程序記憶體 | 移到 Redis Pub/Sub（支援多後端）| `backend/ws_manager.py` |
| 線上人數 | 算連線數 | 視需求改為按使用者/IP 去重 | `backend/ws_manager.py` |
| 資料保留 | 30 天 | 設定分層保留（熱/冷資料）| InfluxDB 設定 |

### 🟢 選配（功能擴充）

| 項目 | 說明 |
|---|---|
| 設備自動註冊 | 訂閱到新設備時 UPSERT 進 PG（目前只 UPDATE）|
| 告警通知 | 接 Email / Line / Webhook（目前只寫 AlertLog）|
| 多語系 / 權限分級 | 不同客戶只看自己的資料（Row-Level Security）|
| 報表匯出 | CSV / PDF 定期報表 |

## 自架 vs 雲端託管的取捨

| 面向 | 雲端託管（Render / InfluxDB Cloud…）| 自架（VM / K8s + EMQX…）|
|---|---|---|
| 初期成本 | 低，按用量 | 高，要先開機器 |
| 維運負擔 | 低（供應商管）| 高（自己管更新/備份/監控）|
| 擴展彈性 | 中（受方案限制）| 高（完全自控）|
| 大規模成本 | 偏高 | 較低（量大時划算）|
| 適合階段 | 階段一～二 | 階段二～三 |

**建議路線**：
1. **階段一**：全用雲端託管，最省心，US$30～50 解決不休眠與安全基線
2. **階段二**：broker 先自架（EMQX），其餘仍託管
3. **階段三**：核心元件自架上 K8s，雲端只留輔助服務

---

## 一句話總結

> 從免費到商用，**最關鍵的兩個改動是「自架 EMQX broker」與「加上認證」**；  
> 成本則隨設備數與觀看人數分階段成長，從每月 US$30 到數千美元不等，  
> 建議依實際客戶成長逐階升級，不要一步到位。
