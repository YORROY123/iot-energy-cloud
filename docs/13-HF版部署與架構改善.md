# 13 — HF 版部署與架構改善（2026-06）

本文件記錄後端從 Render 遷移到 Hugging Face Spaces 的原因、新架構、以及**改善前後的實測成效**。

---

## 1. 為什麼搬

Render 免費方案是「整個 workspace 每月共用 **750 instance 小時**」。
本帳號有兩個常駐免費服務（`iot-energy-cloud`、`line-ai-agent`），
兩個 24 小時跑會用掉約 1460 小時，**每月月中就會被自動停用**。

解法：把 `iot-energy-cloud` 後端搬到另一個免費平台，Render 只留一個服務即可長期免費。
選定 **Hugging Face Spaces**——免費、**免綁卡**、支援 Docker。

## 4. 遇到的問題：跨區延遲造成的吞吐量瓶頸

搬到 HF 後一度出現「12 台設備全離線、最後回報時間一直往後落」。

**根因**：後端訂閱器是單執行緒、逐筆同步處理，每筆訊息要做
「寫 InfluxDB + 更新 Postgres + 查告警規則」共數次網路來回。
舊版後端與 Postgres 同在 Oregon（~1-5ms），但 HF 在 us-east、Postgres 還留在 Oregon，
**每筆都要東西岸跨區來回（~80ms）**，導致：

- 消費者處理速度（~2.5-3 筆/秒）< 生產者速度（4 筆/秒）
- 訊息越積越多，延遲**線性增長**到數分鐘 → 超過 30 秒判離線門檻 → 全離線

實測（3 秒間隔、Postgres 在 Oregon）：延遲每 20 秒增加約 9 秒，持續累積。

---

## 5. 解法與改善成效（實測數據）

### A. 臨時解：降低生產速度（`SIM_INTERVAL` 3→6 秒）
讓生產（2 筆/秒）< 消化能力，立即止血。

| 間隔 | online | 延遲 lag |
|---|---|---|
| 6 秒 | 12 全程 | 6.8 → 12.1 → 3.4 → 7.1 → 4.3 s（震盪、不增長）✅ |

### B. 治本：Postgres 搬到 Neon us-east（與後端、InfluxDB 同區）
資料庫來回從 ~80ms 降到 ~1-5ms，吞吐量翻數十倍，得以調回 **3 秒即時更新**。

> 原 Render Postgres 五張表皆為空（設備清單來自 InfluxDB），故**無資料需遷移**，
> 後端啟動時由 `init_db()` 自動在 Neon 建好 5 張表。

### 改善前後對比（皆為 3 秒間隔 4 筆/秒）

| 指標 | 改善前（Postgres@Oregon） | 改善後（Postgres@Neon us-east） |
|---|---|---|
| 資料延遲 lag | ❌ 無限增長至數分鐘 | ✅ 穩定 **1-4 秒** |
| 線上設備 | ❌ 0 / 12（全離線） | ✅ **12 / 12** |
| 24h 資料量趨勢 | ❌ 持續下降（寫不贏滾出） | ✅ 回升（寫入 > 滾出，完全追上） |
| 看板更新頻率 | — | ✅ 回到 **3 秒即時** |
| 後端↔Postgres RTT | ~80 ms（跨區） | ~1-5 ms（同區） |

實測（3 秒間隔、Neon us-east）：
```
lag: 3.6 → 3.8 → 4.0 → 1.2 → 1.5 → 1.7 s   （平穩、甚至下降）
online: 12 全程
points24h: 206213 → 206225（回升）
```

## 6. 環境變數（HF Space → Settings → Variables and secrets）

Secrets：`INFLUX_TOKEN`、`REDIS_URL`、`POSTGRES_URL`（Neon）、`ADMIN_KEY`
Variables：`DEMO_MODE=false`、`RUN_SIMULATOR=true`、`MQTT_HOST=broker.hivemq.com`、
`MQTT_PORT=1883`、`INFLUX_URL`、`INFLUX_ORG`、`INFLUX_BUCKET=realtime`、`SIM_INTERVAL=3`

部署步驟詳見 [backend/DEPLOY-HF.md](../backend/DEPLOY-HF.md)。
