# 07 — 模擬資料 vs 實際部署的差距

這份文件清楚說明：**目前線上 Demo 的資料是怎麼來的**，以及**真實場域實際上會怎麼做**，兩者差在哪裡。  
看 Demo 的人一定要理解這個差距，才不會誤以為「這就是完整的生產系統」。

---

## 一句話總結

> **Demo 的感測器資料是後端「自己模擬產生」後丟到公開 MQTT broker 再收回來的；
> 真實系統的資料是「實體感測器」透過工業通訊與 4G 上傳到「自架的 MQTT broker」。**
>
> 中間的後端、資料庫、看板邏輯**完全一樣**——差別只在「資料的源頭」與「broker 的等級」。

---

## 資料流對照圖

### 🟡 目前的 Demo（模擬）

```
┌─────────────────────────────────────────────────────────┐
│  Render 後端（單一程式，免費方案無獨立 worker）          │
│                                                           │
│  ┌──────────────────────┐                                │
│  │ 發布執行緒            │  ← random.uniform() 產生假數值 │
│  │ simulator_pub.py     │     例：溫度 = -18.3°C（亂數） │
│  └──────────┬───────────┘                                │
└─────────────│─────────────────────────────────────────────┘
              │ publish（走出去）
              ▼
   ┌──────────────────────────────────┐
   │ broker.hivemq.com（公開、免費）  │  ← 全世界共用，無帳密、無加密
   └──────────────┬───────────────────┘
              │ subscribe（收回來）
              ▼
┌─────────────────────────────────────────────────────────┐
│  Render 後端（同一個程式）                               │
│  ┌──────────────────────┐                                │
│  │ 訂閱執行緒            │ → InfluxDB / PostgreSQL        │
│  │ mqtt_subscriber.py   │ → WebSocket → 看板             │
│  └──────────────────────┘                                │
└─────────────────────────────────────────────────────────┘
```

**重點**：資料是「假的」，但它**真的走了一趟 MQTT 協定**（出去到 hivemq，再訂閱回來），  
所以後端的訂閱、寫庫、告警、推播邏輯跟真實系統一模一樣，只是源頭是模擬的。

### 🟢 真實場域（生產）

```
┌─────────────────────────────────────────────────────────┐
│  現場（門市 / 工廠 / 大樓）                              │
│                                                           │
│  [實體溫度感測器] [實體電表] [實體濕度計] [實體CO₂計]    │

│         ▼                                                 │

│  │ IoT Gateway（邊緣盒） │  ← 真實硬體，讀取感測器暫存器  │

│  └──────────┬───────────┘                                │
└─────────────│─────────────────────────────────────────────┘
              │ publish over TLS（加密）+ 帳密認證
              ▼
   ┌──────────────────────────────────┐
   │ 自架 EMQX 叢集（私有 broker）    │  ← 自己掌控、有認證、有 TLS

   └──────────────┬───────────────────┘
              │ subscribe
              ▼
┌─────────────────────────────────────────────────────────┐
│  後端服務（可水平擴展多台）                             │
│  訂閱 → InfluxDB / PostgreSQL → WebSocket → 看板         │
│  （這一段跟 Demo 完全相同）                              │
└─────────────────────────────────────────────────────────┘
```

## 逐項差距對照表

| 項目 | 🟡 Demo（模擬） | 🟢 真實生產 |
|---|---|---|
| **資料來源** | `random.uniform()` 產生的亂數 | 實體感測器的真實量測值 |
| **感測器連接** | 無（純軟體） | RS485 / Modbus RTU / 4-20mA |
| **資料上傳設備** | 後端的一條執行緒 | 實體 IoT Gateway（邊緣運算盒）+ 4G SIM 卡 |
| **模擬器位置** | 跟後端同一個程式（Render 免費版無 worker） | 散布在各現場的獨立硬體 |
| **MQTT Broker** | `broker.hivemq.com`（公開共用） | 自架 EMQX 叢集（私有） |
| **Broker 認證** | 無（任何人都能連） | Username/Password + Client 憑證 |
| **傳輸加密** | 無（明文 1883） | TLS（8883 port）|
| **Topic 隔離** | 大家共用 `iess/#`（可能撞名） | 每客戶獨立命名空間 + ACL 權限控管 |
| **遠端控制** | 只回 ack，不真的動作（`DEMO_MODE`）| QoS-2 指令真的下發到設備執行 |
| **資料正確性** | 亂數，無物理意義 | 反映真實設備狀態，可據此決策 |
| **異常告警** | 1% 機率亂數注入假異常 | 真實超溫/超載觸發，連動通知 |
| **可靠度** | 免費主機會休眠 | 高可用部署，不中斷 |

## 哪些部分「完全一樣」？

這點很重要——Demo 不是玩具，**後端核心邏輯就是生產級的**：

| 模組 | Demo 與生產是否相同 |
|---|---|
| `mqtt_subscriber.py`（訂閱、解析、寫庫、告警、推播）| ✅ 完全相同 |
| `database.py`（InfluxDB / PostgreSQL / Redis 連線）| ✅ 完全相同 |
| `models.py`（資料表結構）| ✅ 完全相同 |
| `ws_manager.py`（WebSocket 跨執行緒推播）| ✅ 完全相同 |
| `routers/data.py`（查詢 API）| ✅ 完全相同 |
| `dashboard.html`（前端看板）| ✅ 完全相同 |
| 資料的**源頭** | ❌ Demo 是模擬、生產是實體感測器 |
| MQTT broker 的**等級** | ❌ Demo 用公開、生產用自架 |

## 從 Demo 升級到生產，要改什麼？

只需要動「源頭」和「broker」，後端幾乎不用改：

### 1. 把模擬器換成真實 Gateway
- 移除 `RUN_SIMULATOR=true`（不再自己產資料）
- 在現場部署 IoT Gateway，設定它讀取感測器並 publish 到你的 broker
- Topic 格式維持 `iess/{customer_id}/{site_id}/{device_type}/{device_uid}`

### 2. 把公開 broker 換成自架 EMQX
- 用本專案的 `docker-compose.yml` 自架 EMQX（已內建）
- 設定 `MQTT_HOST` 指向你的 EMQX 位址
- 開啟 EMQX 的認證與 TLS（見 [docs/05-技術探討.md](05-技術探討.md#iot-安全性考量)）

### 3. 開啟真實控制
- 把 `DEMO_MODE` 設為 `false`（控制指令會真的 publish 而非只回 ack）
- Gateway 需訂閱 `iess/control/{device_uid}` 並執行收到的指令

### 4. 強化安全與可靠度
- MQTT 改走 TLS（8883）+ 帳密
- 後端、資料庫部署到不會休眠的付費方案或自架伺服器
- 詳見 [docs/05-技術探討.md](05-技術探討.md) 的安全與水平擴展章節

## 為什麼 Demo 要這樣設計？

| 限制 | 原因 |
|---|---|
| 為何不接真實感測器？ | 雲端 Demo 沒有實體硬體可接 |
| 為何模擬器塞在後端裡？ | Render 免費方案不支援獨立 background worker |
| 為何用公開 broker？ | 自架 EMQX 需要不休眠的主機（免費方案會休眠導致斷線）|
| 為何資料仍走一趟 MQTT？ | 為了讓後端的訂閱/告警/推播邏輯**真實運作**，而非造假 |

> 換句話說：**Demo 在「免費 + 無硬體」的限制下，盡可能保留了真實系統的運作邏輯。**
> 唯一刻意模擬的是「資料源頭」，其餘整條處理鏈都是真的。
