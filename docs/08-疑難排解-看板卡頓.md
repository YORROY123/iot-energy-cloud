# 08 — 疑難排解：看板顯示「已連線」卻凍住不動

這是一則真實的除錯記錄（post-mortem），說明一個典型的 WebSocket「半開連線」問題：  
**看板綠點顯示「已連線」，數值卻停在某個時間點不再更新。**

---

## 症狀

- 看板右上角綠點亮著、顯示「已連線」
- 所有設備卡片的時間戳都停在同一個時刻（例如 `下午 2:27:06`）
- 過了好幾分鐘，數值完全沒變化
- 電力趨勢圖也不再延伸

直覺會以為「後端掛了」或「資料沒進來」，但實際上**後端完全正常**。

---

## 診斷過程

### 第 1 步：確認後端是否還活著

直接查 InfluxDB Cloud，看過去 90 秒有沒有新資料：

```flux
from(bucket:"realtime")
  |> range(start:-90s)
  |> filter(fn:(r)=>r._measurement=="sensor_data" and r.device_uid=="fridge01")
  |> count()
```

結果：**90 秒內 30 筆，精準每 3 秒一筆**。

> 結論：MQTT 發布 → broker → 訂閱 → 寫 InfluxDB 這條鏈**完全正常**，後端沒掛。

### 第 2 步：縮小範圍

| 環節 | 狀態 |
|---|---|
| 模擬器發布 MQTT | ✅ 正常 |
| 後端訂閱 + 寫 InfluxDB | ✅ 正常（資料持續進來）|
| 後端 → 瀏覽器 WebSocket 推播 | ❌ 停止 |

問題鎖定在**最後一段：後端推播到瀏覽器的 WebSocket**。

---

## 根本原因

### 原因 1：WebSocket「半開連線」（half-open connection）

Render（以及大多數雲端平台）在後端前面有一層**反向代理 / 負載平衡器**。  
這層代理可能會**悄悄切斷長時間的 WebSocket 連線**，但：

- 後端那側：下次 `send` 時才會發現連線已死
- 瀏覽器那側：**沒有收到 TCP 關閉訊號**，以為連線還在 → 綠點繼續亮「已連線」

這就是「半開連線」：一邊以為還連著，實際上資料根本傳不過去。

```
瀏覽器  ──────────  代理層  ──╳──  後端
（以為還連著）      （已悄悄切斷）   （send 才發現）
   ↑ 綠點還亮著                     ↑ client 被標記為 dead
```

### 原因 2：推播被慢速資料庫拖累（次要）

原本後端收到 MQTT 訊息的處理順序是：

```
收到訊息 → 寫 InfluxDB（慢）→ 寫 PostgreSQL（慢）→ 才推播瀏覽器
```

在 Render 免費版（CPU 僅 0.1 核）上，每筆訊息都要**同步**寫 InfluxDB Cloud（網路往返），  
12 台設備每 3 秒就有 12 次同步寫入，處理速度跟不上 → 推播時快時慢，體感「卡」。

---

## 解法

### 修正 1：看板加「斷線看門狗」（watchdog）

既然資料正常時每 3 秒一定有一筆，那就**監控「最後收到訊息的時間」**：  
若超過 10 秒沒收到任何訊息，判定連線已死 → 主動關閉並重連。

```javascript
const STALE_MS = 10000;     // 超過 10 秒沒資料就重連
let _lastMsgTime = 0;

ws.onmessage = (e) => {
  _lastMsgTime = Date.now();   // 每收到一筆就更新存活時間
  // ... 處理資料 ...
};

ws.onopen = () => {
  _lastMsgTime = Date.now();
  setInterval(() => {
    if (Date.now() - _lastMsgTime > STALE_MS) {
      ws.close();              // 觸發 onclose → 自動重連
    }
  }, 3000);
};

ws.onclose = () => {
  setTimeout(connect, 2000);   // 斷線後 2 秒重連
};
```

> 關鍵：**不要相信 WebSocket 的「連線狀態」，要用「實際資料的新鮮度」來判斷存活。**

### 修正 2：後端把推播移到寫庫之前

讓瀏覽器**第一時間**拿到資料，不被慢速資料庫拖累：

```python
def on_message(client, userdata, msg):
    # 解析訊息 ...

    # ① 先推播給瀏覽器（即時）
    ws_manager.broadcast_from_thread({...})

    # ② 再寫 InfluxDB（慢，但不影響看板即時性）
    write_api.write(...)

    # ③ 再寫 PostgreSQL + 告警判斷
    db.execute(...)
```

---

## 驗證

部署後：

1. 後端 `GET /health` 回傳 `{"status":"ok"}`
2. InfluxDB 持續每 3 秒收到資料
3. 看板強制重整（`Ctrl+F5`）後，數值順暢每 3 秒跳動
4. 手動測試：把後端暫停幾秒再恢復，看板會顯示「連線逾時，重連中…」然後自動恢復

---

## 帶走的經驗

| 教訓 | 說明 |
|---|---|
| 別相信「連線狀態」 | WebSocket 顯示 OPEN 不代表資料傳得過去，要用資料新鮮度判斷 |
| 雲端代理會切長連線 | PaaS 平台的反向代理常會切斷閒置或長時間的 WebSocket |
| 即時推播優先於持久化 | 對「即時看板」這類場景，先推播再寫庫，體感差很多 |
| 先證明後端是好的 | 從資料源頭（InfluxDB）往回查，快速排除「後端掛了」的假設 |
