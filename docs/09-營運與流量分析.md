# 09 — 營運與流量分析

這份文件說明線上 Demo（GitHub Pages + Render 免費版）的**並發承載能力、流量估算、與營運注意事項**，  
給接手的人快速掌握「這套東西能撐多少人、要注意什麼、何時該升級」。

---

## 一、能同時幾個人進來？

整條鏈每一層的並發上限不同，**真正的瓶頸是 Render 免費版的後端**。

| 層 | 並發能力 | 是否瓶頸 |
|---|---|---|
| **GitHub Pages**（看板網頁）| 幾乎無上限（靜態檔，走 CDN）| ❌ |
| **Render 後端**（免費版 0.1 CPU / 512MB）| **約 10～30 人同時順暢** | ✅ **瓶頸** |
| InfluxDB Cloud（免費）| 讀取 300MB / 5 分鐘 | 多人狂查歷史才會碰到 |
| Upstash Redis（免費）| 10,000 指令 / 天 | Demo 幾乎用不到 |
| HiveMQ 公開 broker | 共用、無 SLA | 可能隨時被斷 |

### 白話結論

| 同時人數 | 體感 |
|---|---|
| 10～30 人 | 順暢 |
| 30～100 人 | WebSocket 推播開始延遲、卡頓 |
| > 100 人 | 0.1 CPU 撐不住，可能掉連線 |

> **為什麼卡在 Render？** 免費版只有 0.1 顆 CPU，後端要同時做三件事：
> 每 3 秒寫 12 筆 InfluxDB、廣播即時資料給所有看板、處理歷史查詢。CPU 很快吃滿。

### 怎麼知道現在有多少人看？

- 看板右上角有「👁 線上 N」即時顯示同時在線人數
- 或直接打 API：`GET https://yorroy123-iot-energy-cloud.hf.space/stats` → `{"online": 3}`

## 二、流量（頻寬）估算

| 項目 | 用量 | 免費額度 | 評估 |
|---|---|---|---|
| GitHub Pages 頻寬 | 看板 HTML 約 20KB / 次載入 | 軟上限 100GB/月 | 充裕（可數百萬次載入）|
| Render 對外流量 | 每位看板約 0.5KB/秒（WebSocket）| 100GB/月 | 約 30 人連續 24 小時才接近上限 |
| InfluxDB 寫入 | 每 3 秒 12 筆，量極小 | 5MB / 5 分鐘 | 充裕 |
| InfluxDB 讀取 | 每次歷史查詢約數十 KB | 300MB / 5 分鐘 | 多人頻繁查才需注意 |

### 計算範例

- 30 位觀眾連續在線：`30 × 0.5KB/s × 86400s × 30天 ≈ 39GB/月` → 在 100GB 額度內
- Chart.js 由 jsDelivr CDN 載入，**不計入** GitHub Pages 頻寬

## 三、營運注意事項

### 🔴 安全相關

| 項目 | 現況 | 風險 / 建議 |
|---|---|---|
| API 無認證 | `/api/*` 任何人可呼叫 | Demo 可接受；正式需加 API Key / JWT |
| CORS 全開 | `allow_origins=["*"]` | 任何網站都能呼叫後端 |
| 公開 MQTT broker | `broker.hivemq.com` | 他人可訂閱/發布 `iess/#`，可能看到或灌假資料 |
| Token 安全 | InfluxDB/Redis token 存在 Render 環境變數 | ✅ 未寫入前端；切勿上傳 `專案憑證記錄.md` |

### 🟡 可用性相關

| 項目 | 注意事項 |
|---|---|
| 冷啟動 | 閒置 15 分鐘休眠，下次訪客等約 30 秒 |
| PostgreSQL 90 天到期 | Render 免費 DB 90 天後刪除，需重建並更新 `POSTGRES_URL` |
| InfluxDB 保留 30 天 | 超過 30 天的歷史資料自動消失 |
| 公開 broker 不穩 | HiveMQ 可能無預警斷線（看板已有自動重連）|

### 🟢 監控建議

- **Render 後台 → Metrics**：看 CPU / 記憶體，接近 100% 就是該升級的訊號
- **看板線上人數**：右上角即時顯示，或 `GET /stats`
- **InfluxDB 用量**：InfluxDB Cloud 後台 → Usage 看讀寫量是否接近免費上限

## 四、要承載更多人怎麼辦？

| 需求 | 做法 | 約略月成本 |
|---|---|---|
| 50～200 人順暢 | Render 升級 Starter（0.5 CPU）| US$7 |
| 不要冷啟動 | 升級付費版（不休眠）| US$7 |
| 上千人 | 後端水平擴展 + 自架 EMQX 叢集（見 [docs/05](05-技術探討.md#水平擴展路線圖)）| 視規模 |

### 進一步優化方向（不一定要花錢）

1. **降低發送頻率**：把模擬器 `SIM_INTERVAL` 從 3 秒調成 5～10 秒，減輕 CPU 與 InfluxDB 壓力
2. **InfluxDB 寫入改批次**：目前每筆同步寫，可改用非同步批次 API 降低 CPU 占用
3. **WebSocket 訊息合併**：把 12 台設備的更新合併成一筆推播，減少廣播次數
4. **歷史查詢加快取**：相同區間查詢用 Redis 快取結果，避免重複打 InfluxDB

## 五、關鍵端點速查

| 端點 | 用途 |
|---|---|
| `GET /health` | 健康檢查（含 demo_mode 狀態）|
| `GET /stats` | 目前線上 WebSocket 人數 |
| `GET /api/history?device_uid=X&start=...&end=...` | 歷史查詢（絕對時間）|
| `GET /api/history?device_uid=X&range=24h` | 歷史查詢（相對範圍）|
| `WS /ws/{client_id}` | 即時資料推播 |

## 六、一句話總結

> **這套免費 Demo 適合「給少數人展示」（同時 10～30 人）**。
> 要對外大量開放，先升級 Render 後端（最便宜 US$7/月解決 CPU 與冷啟動），
> 再往上才需要動到自架 broker 與水平擴展。
