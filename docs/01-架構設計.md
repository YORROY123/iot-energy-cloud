## 三層 IoT 架構

IoT 系統通常以感知層 → 網路層 → 應用層三層描述：

```
┌──────────────────────────────────────────────────────────────────┐
│  感知層（Perception Layer）                                        │
│                                                                    │
│  冷凍冷藏溫度 ────┐                                                │
│  電力計 (CT)  ────┤   RS485 / Modbus RTU                          │
│  溫濕度感測器 ────┤──────────────────────→ IoT Gateway             │
│  CO₂ 感測器   ────┘                        (工業邊緣計算盒)         │
│                                                                    │
│  特性：低頻取樣（1～60 秒），感測器使用 485 串列匯流排              │
└─────────────────────────────────┬────────────────────────────────┘
                                  │ MQTT over TLS / 4G LTE
┌─────────────────────────────────▼────────────────────────────────┐
│  網路層（Network Layer）                                           │
│                                                                    │
│  EMQX 5 MQTT Broker                                               │
│  ・支援 MQTT 3.1.1 / 5.0                                          │
│  ・連線數：單機可達 100 萬並發                                      │
│  ・QoS 0 / 1 / 2                                                  │
│  ・叢集擴展（Raft / External 模式）                                │
│                                                                    │
└─────────────────────────────────┬────────────────────────────────┘
                                  │ MQTT subscribe（後端訂閱）
┌─────────────────────────────────▼────────────────────────────────┐
│  應用層（Application Layer）                                       │
│                                                                    │
│  FastAPI Backend                                                   │
│  ├─ MQTT Subscriber（daemon thread）                               │
│  ├─ Alert Engine（閾值檢查、寫 AlertLog）                          │
│  ├─ REST API（查詢、控制、排程）                                    │
│  └─ WebSocket Manager（即時推播至瀏覽器）                           │
│                                                                    │
│  InfluxDB 2.7   PostgreSQL 16   Redis 7                            │
│  （時序資料）    （元資料/告警）   （快取/排程）                    │
│                                                                    │
│  Dashboard（browser）←── WebSocket ─── Backend                    │
└──────────────────────────────────────────────────────────────────┘
```

## 服務元件說明

### EMQX 5（MQTT Broker）

| 屬性 | 值 |
|---|---|
| Image | `emqx:5` |
| MQTT port | `1883` (TCP), `8883` (TLS) |
| WebSocket | `8083` |
| 管理後台 | `:18083` |
| 預設帳密 | `admin` / `admin123` |

EMQX 為零依賴、高吞吐的 Erlang/OTP 系統。本專案使用單節點模式；生產環境可三節點 Raft 叢集。

---

### FastAPI Backend

| 檔案 | 職責 |
|---|---|
| `main.py` | 應用程式進入點、lifespan hook、WebSocket 端點 |
| `ws_manager.py` | 維護所有 WebSocket 連線的 ConnectionManager 單例 |
| `database.py` | SQLAlchemy engine、InfluxDB client、Redis client |
| `models.py` | ORM 模型（Customer, Site, Device, AlertRule, AlertLog）+ Pydantic schemas |
| `mqtt_subscriber.py` | 訂閱 MQTT、解析 Payload、寫 InfluxDB、更新 PG、告警 |
| `routers/data.py` | 查詢類 REST API |
| `routers/control.py` | 控制類 REST API |

---

### InfluxDB 2.7（時序資料庫）

- **Measurement**：`sensor_data`
- **Tags**（索引，低基數）：`customer_id`, `site_id`, `device_type`, `device_uid`
- **Field**：`value` (float)
- **Time**：UNIX seconds（設備端時間戳，非伺服器收到時間）
- **查詢語言**：Flux
- **Bucket TTL**：本機測試無限期，生產建議 90 天

---

### PostgreSQL 16（關聯式資料庫）

儲存**不隨時間變化的元資料**：客戶、門市、設備清單、告警規則、告警日誌。

---

### Redis 7（快取 / 排程儲存）

- Key pattern：`schedule:{device_uid}`
- Value：JSON，包含 `action`、`cron`、`created_at`
- 生產可改用 BullMQ（Node.js）或 Celery Beat（Python）

---

### Simulator（IoT Gateway 模擬器）

- 12 台虛擬設備，每 3 秒送一次
- paho-mqtt v2 API
- 1% 機率注入異常值（冰箱溫度 → -8°C 而非正常 -20°C；電力 → 3000 W）
- 作為本機開發的 MQTT 資料來源替代真實硬體

## 資料流走向

```
Gateway ──MQTT Publish──→ EMQX ──MQTT Message──→ mqtt_subscriber.py
                                                        │
                    ┌───────────────────────────────────┤
                    │                                   │
                    ▼                                   ▼
               InfluxDB                           PostgreSQL
          (sensor_data point)           (UPDATE devices SET is_online)
                                               (INSERT alert_logs)
                    │
                    └──→ ws_manager.broadcast_from_thread()
                                     │
                              asyncio event loop
                                     │
                         ┌───────────▼───────────┐
                         │  WebSocket clients    │
                         │  (dashboard.html)     │
                         └───────────────────────┘
```

**執行緒邊界**：MQTT 的 `on_message` 在 paho 的背景執行緒中執行，而 WebSocket 推播需在 asyncio event loop 中執行。  
解法：`asyncio.run_coroutine_threadsafe(broadcast_coro, loop)` — 在 MQTT 執行緒中安全地投遞協程。

## MQTT Topic 設計

```
iess/{customer_id}/{site_id}/{device_type}/{device_uid}
```

### 範例

```
iess/C001/S001/temperature/fridge01   → {"value": -22.3, "ts": 1750000000}
iess/C001/S001/power/meter01          → {"value": 1240.5, "ts": 1750000001}
iess/C001/S002/humidity/humid01       → {"value": 62.1, "ts": 1750000002}
iess/C002/S003/co2/co2_01             → {"value": 850.0, "ts": 1750000003}
```

### 控制指令（下行）

```
iess/control/{device_uid}   → {"action": "turn_off", "request_id": "uuid", "expire_at": 1750000010}
```

### 為何這樣設計？

1. **階層隔離**：訂閱 `iess/C001/#` 可取得特定客戶所有資料，方便未來多訂閱者分片
2. **自帶元資料**：不需查庫即可從 topic 得知設備歸屬
3. **避免衝突**：`device_uid` 放最後，方便 wildcard 訂閱（`+/+/+/+`）

## 多租戶資料隔離策略

### PostgreSQL 層

Customer → Site → Device 三層外鍵，查詢時永遠帶 `customer_id` 過濾：

```sql
SELECT d.* FROM devices d
JOIN sites s ON d.site_id = s.id
WHERE s.customer_id = :customer_id
```

### InfluxDB 層

所有 Point 都帶 `customer_id` tag。Flux 查詢範例：

```flux
from(bucket: "realtime")
  |> range(start: -1h)
  |> filter(fn: (r) => r["customer_id"] == "C001")
  |> filter(fn: (r) => r["_measurement"] == "sensor_data")
```

### MQTT 層

後端訂閱 `iess/+/+/+/+`（所有客戶）。  
未來可依客戶拆分獨立訂閱者或 Kafka consumer group。

## 資料庫 Schema

```sql
CREATE TABLE customers (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR NOT NULL,
    email       VARCHAR UNIQUE NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE sites (
    id          SERIAL PRIMARY KEY,
    customer_id INTEGER REFERENCES customers(id),
    name        VARCHAR NOT NULL,
    location    VARCHAR,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE devices (
    id          SERIAL PRIMARY KEY,
    site_id     INTEGER REFERENCES sites(id),
    device_type VARCHAR NOT NULL,      -- temperature / power / humidity / co2
    device_uid  VARCHAR UNIQUE NOT NULL,
    name        VARCHAR,
    is_online   BOOLEAN DEFAULT FALSE,
    last_seen   TIMESTAMPTZ,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE alert_rules (
    id            SERIAL PRIMARY KEY,
    device_uid    VARCHAR NOT NULL,
    condition     VARCHAR NOT NULL,    -- 'gt' | 'lt'
    threshold     FLOAT NOT NULL,
    notify_email  VARCHAR,
    is_active     BOOLEAN DEFAULT TRUE
);

CREATE TABLE alert_logs (
    id           SERIAL PRIMARY KEY,
    rule_id      INTEGER REFERENCES alert_rules(id),
    device_uid   VARCHAR NOT NULL,
    value        FLOAT NOT NULL,
    triggered_at TIMESTAMPTZ NOT NULL,
    resolved_at  TIMESTAMPTZ
);
```

## 並發模型：執行緒 vs async

```
┌──────────────────────────────────────────────────────────────┐
│  asyncio event loop（主執行緒）                               │
│                                                              │
│  ┌──────────┐  ┌──────────┐  ┌────────────────────────────┐ │
│  │ REST API │  │ WebSocket│  │ ws_manager.broadcast()     │ │
│  │ handlers │  │ endpoint │  │ (coroutine, runs here)     │ │
│  └──────────┘  └──────────┘  └─────────────▲──────────────┘ │
│                                             │                │
└─────────────────────────────────────────────│────────────────┘
                                              │ run_coroutine_threadsafe()
┌─────────────────────────────────────────────│────────────────┐
│  paho-mqtt 執行緒（daemon thread）           │                │
│                                             │                │
│  on_message() ──寫 InfluxDB──寫 PG──────────┘                │
│  （blocking I/O 可以直接呼叫，無需 await）                     │
└──────────────────────────────────────────────────────────────┘
```

**關鍵設計**：`ws_manager.set_loop(loop)` 在 FastAPI lifespan 啟動時呼叫一次，  
儲存 asyncio event loop 的參照。MQTT 執行緒呼叫 `broadcast_from_thread()` 時使用此參照跨執行緒投遞。
